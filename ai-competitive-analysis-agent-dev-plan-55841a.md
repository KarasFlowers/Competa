# AI 竞品分析 Agent 协作系统初步开发计划

本计划面向从零开发一个可演示、可追溯、具备真实反馈闭环的 AI 竞品分析多 Agent 协作系统。

## 1. 开发目标

- **核心目标**：实现“信息采集 → 结构化分析 → 报告撰写 → 质检反馈 → 报告输出”的端到端链路。
- **演示目标**：用户输入行业、目标产品和竞品后，系统自动生成带引用来源的结构化竞品分析报告。
- **评分目标**：优先覆盖多 Agent 协作、结构化 Schema、真实反馈闭环、信息溯源、可观测性这几个高权重点。

## 2. 推荐技术路线

- **后端**：Python + FastAPI，负责任务创建、Agent 编排、报告接口、日志与溯源查询。
- **Agent 编排**：LangGraph，显式表达 DAG 流程和质检打回路径。
- **结构化输出**：Pydantic Schema，约束功能树、定价模型、用户画像、SWOT、报告段落和引用证据。
- **存储**：MVP 阶段使用 SQLite，后续可升级 PostgreSQL；保存任务、来源、结构化产物、报告、Trace 日志。
- **前端**：React + TypeScript + Vite + TailwindCSS，提供任务创建、流程查看、报告查看、溯源跳转、Agent 决策回放。
- **模型接入**：通过统一 LLM Adapter 抽象模型供应商，避免业务逻辑绑定具体模型。

## 2.1 Reference 参考取舍

- **采用 LangGraph 思路**：使用 `StateGraph`、条件边和 checkpoint 思路表达可追溯 DAG，重点服务 QA 打回闭环。
- **采用 CrewAI 思路**：借鉴角色化 Agent 定义、任务边界和 `output_pydantic` 风格的结构化输出约束。
- **采用 OpenAI Agents SDK 思路**：借鉴 handoff、guardrails、tracing、session 的概念，但不直接绑定该 SDK。
- **采用 Conductor 思路**：借鉴确定性路由和 Dashboard/SSE 可视化方向，避免让 LLM 决定所有流程分支。
- **采用 ADK 思路**：借鉴图工作流状态管理、断点恢复和子工作流设计，作为后续增强项。
- **采用 GPT-Researcher 思路**：借鉴多源搜索、引用管理和结构化研究报告经验，用于强化信息采集与溯源。
- **不机械照搬**：不复制参考项目源码，不一次性引入过多框架；所有参考点必须服务本项目的竞品分析、溯源、质检闭环和演示目标。

## 3. MVP 功能边界

- **信息采集 Agent**：支持用户输入 URL、文本资料、访谈记录、问卷样本；MVP 可先做“公开网页/手动资料输入 + 模拟问卷/访谈样本”。
- **分析师 Agent**：抽取功能树、定价模型、用户画像，生成竞品对比表和 SWOT。
- **报告撰写 Agent**：基于结构化分析结果生成 Markdown/HTML 报告。
- **质检 Agent**：检查字段完整性、引用完整性、结论是否有来源、Schema 是否合规；不合格时返回明确修复指令。
- **反馈闭环**：至少支持一次真实打回，例如“缺少引用/字段为空/竞品覆盖不足”触发采集或分析重跑。
- **可观测性**：记录每个 Agent 的输入、输出、Prompt、状态、耗时、Token 估算、错误和重试信息。
- **上下文管理**：Agent 间传递结构化摘要而非原始全文；超长输入分片处理；每个 Agent 的上下文窗口有 token 预算上限，超出时自动裁剪低优先度内容。
- **业务 KPI**：定义并追踪覆盖率（信息源数量/预设目标）、引用完整率（有 Evidence 的 Claim 占比）、人工修正率（用户修改次数/总结论数），作为 QA 判定和迭代改进的量化依据。

## 4. 系统模块拆分

- **任务模块**：创建分析任务，定义目标产品、竞品、行业、资料来源和分析范围。
- **Schema 模块**：定义 Source、Evidence、Claim、FeatureTree、PricingModel、Persona、SWOT、Report、AgentMessage、TraceEvent。
- **Agent 模块**：按采集、分析、撰写、质检拆分职责，每个 Agent 只读写标准 Schema。
- **编排模块**：使用 LangGraph 定义 DAG：Start → Collect → Analyze → Write → QA → End，并支持 QA → Collect/Analyze/Write 的条件回退。
- **溯源模块**：所有 Claim 必须绑定 Evidence；报告中每个关键结论展示来源链接或原始片段。
- **API 模块**：提供任务创建、运行状态、报告详情、溯源详情、Trace 列表、人工修正入口。
- **前端模块**：任务表单、Agent 流程可视化、报告页面、证据侧栏、日志/决策回放页面。
- **文档模块**：README、架构图、Agent 角色说明、Schema 协议说明、部署说明、演示脚本。

## 5. 分阶段里程碑

### 阶段一：项目骨架与基础协议

- 初始化前后端项目结构和开发规范。
- 定义核心 Pydantic Schema 与 AgentMessage 协议。
- 实现基础 API：创建任务、查询任务、读取报告、读取 Trace。
- 建立最小数据库模型和本地运行说明。
- 建立 Guardrails 基础层：Agent 输出必须先通过 Schema 校验，再进入下一节点。

### 阶段二：Agent DAG 主链路

- 实现采集、分析、撰写、质检四类 Agent 的最小可用版本。
- 用 LangGraph 串联 DAG 主流程。
- 实现结构化输出校验，失败时返回明确错误。
- 生成第一版可展示的结构化报告。
- 采用确定性路由：常规流程和 QA 分支由代码控制，LLM 只负责专职任务内容生成。

### 阶段三：真实反馈闭环与溯源

- 实现 QA 规则：字段完整性、引用完整性、Schema 合规性、竞品覆盖度。
- 实现 QA 打回 Collect/Analyze/Write 的条件分支。
- 对每条关键结论强制绑定 Evidence。
- 前端支持报告引用标注与溯源详情查看。
- 设计 handoff 风格的结构化返工指令，包含目标 Agent、问题类型、失败字段、证据要求和重试上限。

### 阶段四：可观测性与产品体验

- 完善 Trace 日志：Prompt、输入、输出、状态、耗时、Token、重试、错误。
- 前端展示 Agent 执行时间线、DAG 状态和决策回放。
- 增加人工修正入口，允许用户补充资料或修正结论后重新生成。
- 提供演示数据集，确保无外网或模型波动时仍可演示核心流程。
- 引入 SSE 或轮询式运行状态更新，前端可展示每个 Agent 的实时进度和中间产物。

### 阶段五：工程完善与答辩材料

- 补齐异常处理、超时重试、降级策略和基础测试。
- 完成 README、架构图、Agent 协议文档、部署说明。
- 准备演示脚本、评分点对照表和答辩材料。
- 检查数据合规、隐私脱敏和外部信息来源声明。
- 总结 reference 借鉴点与本项目差异化：垂直竞品分析、强制溯源、QA 闭环、可观测决策回放。

## 6. 优先级建议

- **第一优先级**：端到端流程跑通，确保现场能创建任务并生成报告。
- **第二优先级**：Schema 约束、引用证据和 QA 打回闭环，直接对应最高评分维度。
- **第三优先级**：可观测性界面和决策回放，提升技术深度与产品可信度。
- **第四优先级**：高级采集能力、动态 Schema 演化等远期创新点。
- **MVP 内必实现的前瞻性创新**：QA 打回时自动将失败模式转化为后续 Agent 的约束规则（棘轮机制），即错误不会重复发生，这是现有参考框架均未提供的独特能力。

## 7. 主要风险与应对

- **LLM 输出不稳定**：使用 Pydantic 校验、结构化重试、失败降级模板。
- **上下文溢出**：Agent 间只传递结构化摘要，每个 Agent 有 token 预算，超出时按优先级裁剪。
- **引用缺失或幻觉**：要求所有 Claim 绑定 Evidence，无引用结论不得进入最终报告。
- **反馈闭环流于形式**：设计可触发的 QA 规则，并在演示样例中保留一次打回和改进记录。
- **演示依赖外网**：准备本地样例资料和离线 mock 数据。
- **范围过大**：MVP 先支持少量竞品和有限数据源，再逐步扩展问卷、访谈和网页采集。
- **参考项目过多导致失焦**：只保留可直接提升评分点的参考能力，优先实现本项目自己的最小闭环。

## 8. 验收标准

- **功能验收**：输入目标产品和竞品后，系统能生成结构化竞品报告。
- **协作验收**：至少四类 Agent 职责清晰，DAG 流程和消息传递可查看。
- **闭环验收**：QA 能识别至少一种真实问题并触发重跑，重跑结果有改善记录。
- **溯源验收**：报告关键结论均可定位到 URL、文档片段或访谈/问卷记录。
- **观测验收**：每个 Agent 的输入、输出、Prompt、执行状态和中间产物可追溯。
- **材料验收**：README、架构图、Schema 文档、部署说明和演示脚本齐全。

## 9. 建议下一步

- 确认技术栈是否采用 Python FastAPI + LangGraph + React。
- 确认 MVP 首批支持的数据源范围。
