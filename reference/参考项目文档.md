## 🔴 核心参考项目

### 1. Google ADK (Agent Development Kit)

**本地路径**: `adk-python/`
**仓库**: https://github.com/google/adk-python
**协议**: Apache 2.0
**Python**: 3.11+

#### 简介

Google 官方开源的 Agent 开发框架，提供图结构工作流编排（Workflow Runtime）、Agent 委派（Task API）和完整的评估/部署工具链。支持 fan-out/fan-in、循环、重试、状态管理、human-in-the-loop 和嵌套工作流。

#### 对 CompetitorLens 的参考价值

| 我们的模块 | ADK 对应实现 | 借鉴点 |
|-----------|-------------|--------|
| `DAGOrchestrator` | `Workflow` 图编排引擎 | 拓扑排序 + 并行执行的实现细节，状态管理机制 |
| Agent 间数据传递 | `SharedMemory` / Workflow state | 节点间上下文传递的结构化方式 |
| 工具注册 | `Tool` 注册机制 | MCP 风格的工具定义与权限控制 |
| 任务委派 | `Task API` (单轮/多轮) | Agent 间委派的接口设计 |

#### 关键代码路径

- 工作流引擎: `google/adk/workflows/`
- Agent 定义: `google/adk/agents/`
- 图编排: `google/adk/flow_engine/`

#### 可借鉴的改进

1. **Workflow Runtime 的状态持久化** — ADK 支持断点恢复，我们的 DAG 执行失败后需要完全重跑
2. **嵌套工作流** — 支持在 Workflow 节点中嵌入子 Workflow，可以用来实现 Collector Agent 内部的子任务分解
3. **内置评估工具** — ADK 有 `Eval` 模块，可以帮助我们量化 Agent 输出质量

### 2. Microsoft Conductor

**本地路径**: `conductor/`
**仓库**: https://github.com/microsoft/conductor
**协议**: MIT
**特点**: YAML 驱动、确定性路由、零 token 编排

#### 简介

微软开源的 CLI 工具，用 YAML 定义多 Agent 工作流。路由使用 Jinja2 模板和表达式求值，编排层不消耗任何 token。支持并行执行、子工作流组合、脚本步骤、对话模式和 Web Dashboard。

#### 对 CompetitorLens 的参考价值

| 我们的模块 | Conductor 对应实现 | 借鉴点 |
|-----------|-------------------|--------|
| DAG 编排 | YAML 工作流定义 | 声明式定义 vs 代码定义的优劣对比 |
| SSE 实时推送 | Web Dashboard | 实时 DAG 可视化 + 流式输出 + 交互式 human gate |
| 条件路由 | Jinja2 模板路由 | 零 token 的确定性路由，降低延迟和成本 |
| Agent 间通信 | input_mapping | 结构化的数据传递协议 |

#### 关键代码路径

- 工作流引擎: `conductor/core/`
- YAML 解析: `conductor/core/workflow.py`
- Web Dashboard: `conductor/web/`
- Agent 执行: `conductor/core/agents/`

#### 可借鉴的改进

1. **Web Dashboard 设计** — 实时 DAG 图 + 面包屑导航 + 流式输出 + 浏览器内 human gate，比我们的 TracePanel 更完善
2. **确定性路由** — 编排层零 token 消耗，降低成本，可以用于 Collector Agent 的分发逻辑
3. **子工作流组合** — `for_each` 动态 fan-out + 可复用子工作流，适合 Collector Agent 的并行采集模式
4. **验证机制** — 运行前捕获模板引用过期、缺少输入、未声明依赖等问题

### 3. CrewAI

**本地路径**: `crewAI/`
**仓库**: https://github.com/crewAIInc/crewAI
**协议**: MIT
**Stars**: 44.6k

#### 简介

最流行的多 Agent 角色扮演框架。每个 Agent 有 Role/Goal/Backstory 定义，支持结构化输出（`output_pydantic`）、Agent 间 handoff、Flows 企业级编排。完全独立于 LangChain。

#### 对 CompetitorLens 的参考价值

| 我们的模块 | CrewAI 对应实现 | 借鉴点 |
|-----------|----------------|--------|
| 6 个 Agent 角色 | `Agent(role, goal, backstory)` | 角色定义的结构化方式 |
| 结构化输出 | `output_pydantic` / `output_json` | 强制 Agent 输出符合 Pydantic Schema |
| Agent 间委派 | `handoff` 机制 | Agent 自主委派给其他 Agent 的协议 |
| 任务依赖 | `Task(context=[...])` | 显式声明任务间的数据依赖 |
| 可观测性 | Tracing & Observability | 内置的 Agent 追踪和指标监控 |

#### 关键代码路径

- Agent 定义: `crewai/agent.py`
- Task 定义: `crewai/task.py`
- Crew 编排: `crewai/crew.py`
- Flow 编排: `crewai/flow/`
- 结构化输出: `crewai/utilities/pydantic_schema.py`

#### 可借鉴的改进

1. **`output_pydantic` 强制 Schema** — 确保每个 Agent 的输出都符合预定义 Schema，比我们当前的自由格式输出更可靠
2. **Agent memory** — CrewAI Agent 有短期/长期记忆，可以参考实现 Agent 间更丰富的上下文共享
3. **Guardrails** — 内置输出校验机制，可以作为我们 Governance Layer 的补充
4. **Flows 企业架构** — 事件驱动 + 粒度控制，适合扩展我们的系统支持更复杂的企业场景

### 4. LangGraph

**本地路径**: `langgraph/`
**仓库**: https://github.com/langchain-ai/langgraph
**协议**: MIT
**注意**: `competitorsmart` 子项目已在使用

#### 简介

LangChain 团队出品的低层级 Agent 编排框架。支持有状态工作流、持久化执行（durable execution）、human-in-the-loop、综合记忆管理和 LangSmith 调试。核心概念：图（Graph）+ 状态（State）+ 节点（Node）。

#### 对 CompetitorLens 的参考价值

| 我们的模块 | LangGraph 对应实现 | 借鉴点 |
|-----------|-------------------|--------|
| DAG 执行 | `StateGraph` | 声明式图定义 + 条件边 |
| 状态管理 | `TypedDict` State | 类型安全的状态定义与传递 |
| 持久化 | `MemorySaver` / `SqliteSaver` | 断点恢复，解决我们内存存储的问题 |
| Human-in-the-loop | `interrupt_before` / `interrupt_after` | 审查节点的人工介入机制 |
| 可观测性 | LangSmith 追踪 | 深度调试和性能分析 |

#### 关键代码路径

- 图编排核心: `libs/langgraph/langgraph/graph/`
- 状态管理: `libs/langgraph/langgraph/graph/state.py`
- 持久化: `libs/langgraph/langgraph/checkpoint/`
- 消息处理: `libs/langgraph/langgraph/messages/`

#### 可借鉴的改进

1. **`StateGraph` 声明式定义** — 比我们手写 DAG 更简洁，且支持条件分支
2. **Checkpoint 持久化** — 我们 Planned 的 PostgreSQL 持久化可以参考 LangGraph 的 checkpoint 模式
3. **`interrupt` 机制** — Reviewer Agent 审查时可以中断流程等待人工确认，实现真正的 human-in-the-loop
4. **子图（Subgraph）** — Collector Agent 可以实现为子图，内部有自己的状态和节点

### 6. OpenAI Agents SDK

**本地路径**: `openai-agents-python/`
**仓库**: https://github.com/openai/openai-agents-python
**协议**: MIT

#### 简介

OpenAI 官方的轻量级多 Agent 框架。支持 Guardrails（输入输出校验）、Handoffs（Agent 间委派）、Tracing（内置追踪）、Sessions（会话管理）和 Sandbox Agents（沙箱执行）。

#### 对 CompetitorLens 的参考价值

| 我们的模块 | OpenAI Agents 对应实现 | 借鉴点 |
|-----------|----------------------|--------|
| Governance Layer | `Guardrails` | 输入/输出校验的实现方式 |
| Agent 委派 | `Handoffs` | Agent 间委派的协议设计 |
| 可观测性 | `Tracing` | 内置的运行追踪机制 |
| 共享记忆 | `Sessions` | 跨 Agent 运行的会话管理 |

#### 关键代码路径

- Guardrails: `src/agents/guardrail/`
- Handoffs: `src/agents/handoff/`
- Tracing: `src/agents/tracing/`
- Sessions: `src/agents/sessions/`

#### 可借鉴的改进

1. **Guardrails 输入/输出校验** — 可以在我们每个 Agent 的输出端加校验层，确保符合 Schema
2. **Handoff 协议** — 结构化的 Agent 间委派，比我们当前的 SharedMemory 更规范
3. **Sandbox Agents** — 长时间运行的 Agent 在沙箱中执行，增强安全性

## 🟡 竞品参考项目

### 7. DeerFlow (字节跳动)

**本地路径**: `deer-flow/`
**仓库**: https://github.com/bytedance/deer-flow
**协议**: MIT
**⚠️ 字节出品 — 评委可能熟悉**

#### 简介

字节跳动开源的深度研究执行引擎。将大语言模型与专业工具和 Web 搜索集成，实现从研究到成品的完整工作流：多步骤规划 → 自动执行 → 代码生成 → 结果验证 → 报告输出。

#### 对 CompetitorLens 的参考价值

- **研究工作流** — Research → Analysis → Report 的流程设计，和我们的竞品分析流程高度相似
- **多步骤规划** — 将复杂研究任务分解为可执行步骤
- **工具集成** — 搜索引擎 + 代码执行 + 文件操作的集成方式
- **字节生态** — 展示对字节开源生态的了解，答辩时可提及

#### 关键代码路径

- 工作流核心: `src/graph/`
- Agent 定义: `src/agents/`
- 工具集成: `src/tools/`

#### 差异化方向

我们 vs DeerFlow：
- 我们专注**竞品分析**（垂直场景），DeerFlow 是通用深度研究
- 我们有 **6 专职 Agent + DAG 编排**，DeerFlow 更偏向单 Agent 多步骤
- 我们有 **Harness 五层架构 + Ratchet 机制**，这是 DeerFlow 没有的

### 8. GPT-Researcher

**本地路径**: `gpt-researcher/`
**仓库**: https://github.com/assafelovic/gpt-researcher
**协议**: MIT
**Stars**: 16k+

#### 简介

自主深度研究 Agent。给定研究主题，自动搜索多个信息源、抓取网页内容、聚合分析、生成带引用的结构化研究报告。支持并行搜索和多模型后端。

#### 对 CompetitorLens 的参考价值

| 我们的模块 | GPT-Researcher 对应实现 | 借鉴点 |
|-----------|----------------------|--------|
| Collector Agent | `WebSearchTool` + 并行搜索 | 多源搜索策略和去重算法 |
| Citation Agent | 内置引用系统 | 自动生成引用 + 来源验证 |
| 报告生成 | `ReportGenerator` | Markdown 报告的结构化生成 |

#### 关键代码路径

- 搜索引擎: `backend/search/`
- 报告生成: `backend/report/`
- Agent 核心: `backend/agent/`

#### 差异化方向

我们 vs GPT-Researcher：
- 我们是**多 Agent 协作**（6 Agent），GPT-Researcher 是**单 Agent 多工具**
- 我们有 **DAG 编排 + 反馈闭环**，GPT-Researcher 是线性流程
- 我们有 **Harness 五层架构 + 棘轮机制**，提供更强的工程化保障

## 🎯 建议的改进路线

基于参考项目的分析，建议按以下优先级改进 CompetitorLens：

### P0 — 立即可做（答辩加分）

1. **添加 Guardrails 层** — 参考 OpenAI Agents SDK，在每个 Agent 输出端加 Schema 校验
2. **参考 ADK 的 Workflow 定义** — 用更声明式的方式重写 DAG 创建逻辑
3. **参考 Conductor 的 Web Dashboard** — 增强前端的实时 DAG 可视化

### P1 — 核心改进（提升完成度）

4. **实现持久化存储** — 参考 LangGraph 的 Checkpoint 模式，实现 PostgreSQL 持久化
5. **完善反馈闭环** — 参考 CrewAI 的 handoff + LangGraph 的条件边，让 Reviewer→Writer 闭环真正可用
6. **添加 human-in-the-loop** — 参考 LangGraph 的 interrupt 机制，在关键节点支持人工介入

### P2 — 长期优化

7. **结构化 Agent 输出** — 参考 CrewAI 的 `output_pydantic`，强制所有 Agent 输出符合 Schema
8. **嵌套子图** — 参考 ADK 的子 Workflow，让 Collector Agent 支持内部子任务分解
9. **评估体系** — 参考 ADK 的 Eval 模块，建立 Agent 输出质量的量化评估

## 📝 答辩中如何提及这些项目

> "我们的系统参考了 Google ADK 的图编排思想、Conductor 的声明式工作流定义，以及 CrewAI 的角色化 Agent 设计。但我们没有直接使用这些框架，而是自研了 Harness 五层架构，因为我们发现现有框架在**竞品分析这个垂直场景**中缺少几个关键能力：信息溯源的强制约束、错误自动转化为永久约束的棘轮机制，以及质检 Agent 对分析结论的独立验证。"
